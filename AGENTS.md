# AGENTS.md

## 저장소 목적
이 저장소는 대화형 TUI 기반 연구 에이전트 시스템을 구현하고 검증한다.

항상 다음을 우선한다.
- 실제 인터랙티브 동작의 정확성
- 상태 일관성
- 재현 가능한 검증
- 근거에 기반한 과학적 산출물

겉보기에 그럴듯한 동작, 보기 좋은 데모, 매끈한 문장을 위해 정확성, 추적 가능성, 재현성을 희생하지 않는다.

## 작업 방식
- 하나의 일관된 이슈 또는 검증 대상마다 하나의 작업 흐름을 유지한다.
- 복잡한 버그나 단순하지 않은 기능 작업은 수정 전에 먼저 계획을 세운다.
- 큰 폭의 리팩터링보다, 작고 확실한 수정안을 우선한다.
- 관련 검증 흐름을 다시 실행하기 전에는 버그가 해결되었다고 판단하지 않는다.
- 근거의 완전성과 내부 정합성을 확인하기 전에는 원고가 완성되었다고 판단하지 않는다.
- 메인 에이전트는 요구사항 파악, 의사결정, 최종 종합에 집중한다.
- 서브에이전트는 범위가 제한된 탐색, 로그 분석, 테스트 초안 작성, 파일 단위 조사, 근거 점검, 정합성 검토에 활용한다.
- 쓰기 비중이 큰 작업에서는 최종 수정 경로를 하나의 오케스트레이터 아래로 모은다.

## TUI 검증 원칙
- TUI의 실제 동작은 단순한 단위 테스트보다 우선하는 중요한 진실의 원천으로 취급한다.
- 라이브 검증에서 버그가 확인되면 반드시 다음을 기록한다.
  - 재현 절차
  - 기대 동작
  - 실제 동작
  - 새 세션에서도 재현되는지 여부
  - 저장된 아티팩트와 in-memory/UI 상태가 어긋나는지 여부
- 아래 유형을 명확히 구분해서 판단한다.
  - persisted state bug
  - in-memory projection bug
  - refresh/render bug
  - session resume/reload bug
  - race/timing bug

## 인터랙티브 버그 처리 절차
1. 먼저 이슈를 재현한다.
2. 패치 전에 짧고 명확한 버그 기록을 남긴다.
3. 관련 가능성이 높은 하위 시스템을 식별한다. 예: renderer, projection, persistence, refresh loop, resume path, event/state synchronization
4. 문제가 단순하지 않다면 수정 전에 짧은 계획을 작성한다.
5. 증상을 설명할 수 있는 가장 작은 수정부터 적용한다.
6. 동일한 검증 흐름을 다시 실행한다.
7. 인접한 인터랙티브 흐름에 회귀가 없는지 확인한다.

## 검증 완료 기준
다음이 모두 충족되기 전까지 작업은 끝난 것이 아니다.
- 기존 버그가 더 이상 재현되지 않는다
- 관련 검증 또는 스모크 플로우를 다시 실행했다
- 중요한 로그와 아티팩트의 일관성을 확인했다
- 인접 영역의 명백한 회귀 가능성을 검토했다
- 최종 요약에 원인, 수정 내용, 남은 리스크가 설명되어 있다

## 변경 원칙
- 작업이 명시적으로 기존 계약 변경을 요구하지 않는 한, 기존 계약을 유지한다.
- 추측에 기반한 대규모 재작성은 피한다.
- 수정 목적에 포함되지 않은 UX 동작을 조용히 바꾸지 않는다.
- 불확실한 부분은 숨기지 말고 가정으로 명시한다.

## 과학적 글쓰기 및 원고 생성 원칙
이 저장소는 과학 논문 원고 또는 논문에 준하는 연구 산출물을 생성할 수 있다.
문장이 유창해 보이는 것보다, 근거의 밀도, 내부 정합성, 재현 가능성을 더 중요하게 본다.

### 핵심 원칙
원고는 현재 확보된 근거가 뒷받침하는 수준을 넘는 주장을 해서는 안 된다.

다음은 금지한다.
- 누락된 방법 세부사항을 지어내는 일
- 데이터셋 속성을 추측으로 채우는 일
- 통계 검정이나 신뢰구간을 임의로 만들어내는 일
- 뒷받침 아티팩트 없이 재현성을 주장하는 일
- 약한 근거를 강한 실증 결과처럼 서술하는 일

근거가 부족하면 다음 중 하나를 택한다.
- 주장의 표현 수위를 낮춘다
- 결과물을 미완성 또는 blocked 상태로 표시한다
- 문서의 성격을 더 약하고 정직한 형태로 낮춘다

## 주장-근거 계약
중요한 과학적 주장은 모두 명시적인 아티팩트나 계산 가능한 근거에 의해 뒷받침되어야 한다.

특히 아래 유형의 주장은 반드시 근거가 필요하다.
- 성능 향상 주장
- 안정성 또는 강건성 주장
- 재현성 주장
- 효율성 또는 자원 사용 관련 주장
- 새로움 또는 선행연구 대비 차별성 주장

근거가 부족하면 자동으로 더 약한 표현을 사용한다.
예:
- “demonstrates improvement” → “suggests a small positive delta under this benchmark”
- “significant improvement” → 통계적 근거 없이는 사용 금지
- “reproducibility requirement satisfied” → 뒷받침 아티팩트 없이는 사용 금지

## Method 완전성 기준
다음 항목이 필요한 상황에서 빠져 있으면 원고는 미완성으로 본다.
- 데이터셋 이름
- 데이터셋 출처
- 샘플 수
- feature 수
- class 수
- 클래스 불균형 또는 missingness 관련 정보
- 전처리 단계와 순서
- 전처리가 각 fold 내부에서만 fit되는지 여부
- outer CV fold 수
- inner CV fold 수
- 반복 횟수
- stratification 정책
- random seed 또는 seed 목록
- 모델 하이퍼파라미터 탐색 공간
- 모델 선택 metric
- outer reporting metric
- runtime 측정 방식
- memory 측정 방식

알 수 없는 값을 추측으로 채워 넣지 않는다.
필수 Method 정보가 코드나 아티팩트에서 확인되지 않으면, 원고 완전성 검사를 실패시키거나 blocked 상태로 표시한다.

## Results 완전성 기준
원고가 aggregate summary metric만 제시하고 더 세분화된 근거를 제공하지 않으면 미완성으로 본다.

적용 가능한 경우 최소한 다음을 포함해야 한다.
- 데이터셋별 결과
- std 등 분산 또는 변동성 추정치
- 신뢰구간, 또는 이를 제공할 수 없는 이유에 대한 명시적 설명
- 반복 평가를 주장하는 경우 paired 또는 repeated comparison 아티팩트
- 정보가 담긴 캡션을 가진 최소 1개의 과학적 figure

권장 결과 아티팩트 예시:
- dataset-level result table
- paired-difference summary
- aggregate summary table
- forest plot
- paired-difference boxplot
- runtime/memory tradeoff plot

다음과 같은 결과 표현은 사용하지 않는다.
- pass/fail 형식의 figure
- 코드 변수명이 그대로 드러나는 캡션
- 내부 JSON key나 logging 문구를 그대로 사용한 캡션

## 통계적 서술 원칙
관측된 향상이 변동성에 비해 작다면, 적절한 근거 없이 이를 확립된 개선으로 서술하지 않는다.

- delta > 0이라는 이유만으로 충분한 증거로 간주하지 않는다.
- 효과 크기나 불확실성을 함께 설명해야 하는 상황에서 p-value만으로 결론을 내리지 않는다.
- 통계 근거가 제한적이면 그 한계를 명시하고 주장 수위를 낮춘다.
- repeated CV를 사용했다는 사실만으로 강한 추론적 표현이 정당화되지는 않는다.

## 정합성 lint 기준
생성된 원고는 정합성 검사를 통과하기 전까지 완성으로 간주하지 않는다.

다음 경우에는 결과물을 실패 또는 차단 처리한다.
- Method에 없는 모델이 Results나 Discussion에 등장하는 경우
- 초록, 결과, 결론의 핵심 수치가 서로 일치하지 않는 경우
- 캡션에 내부 변수명, metric key, logging text가 드러나는 경우
- 재현성을 주장하면서 뒷받침 아티팩트가 없는 경우
- 인용된 선행연구 대비 새로움을 과장하는 경우
- 구체적 근거 대신 반복적인 수사적 형용사에 의존하는 경우

정당한 근거 없이 남용하지 말아야 할 표현 예시:
- lightweight
- reproducible
- practical
- modest
- robust

## 관련연구 및 문서 포지셔닝 원칙
근거가 충분하지 않다면 단순 benchmark 요약을 완전한 연구 기여처럼 포장하지 않는다.

적용 가능한 경우 관련연구는 보통 다음 축을 다뤄야 한다.
- 고전적 benchmark 맥락
- nested CV, leakage-safe preprocessing 등 평가 방법론
- resource-aware 또는 lightweight evaluation 맥락
- 데이터셋 또는 태스크별 선행연구

원고에는 가장 가까운 선행연구를 명시하고, 범위 또는 기여 측면에서 어떤 차이가 있는지 분명히 설명해야 한다.

새로움이나 근거 수준이 제한적이라면 다음과 같은 더 정직한 문서 유형을 우선 검토한다.
- benchmark note
- technical report
- reproducibility note

작업 범위가 그 수준에 머무르는데도 억지로 research paper 형식을 강요하지 않는다.

## 재현성 아티팩트 기준
구체적인 재실행 지원이 없으면 재현성을 주장하지 않는다.

필요할 수 있는 아티팩트 예시는 다음과 같다.
- raw results CSV 또는 JSON
- 데이터셋별 metric
- seed 목록
- 환경 명세
- Python 및 주요 라이브러리 버전
- 하드웨어 요약
- figure 생성 스크립트
- 한 줄 재실행 명령

이런 자료가 없다면 재현성을 단정하지 말고, 그 한계를 명시한다.

## 멀티에이전트 사용 원칙
서브에이전트는 범위가 제한되고 검토 가능한 작업에 사용한다.
예:
- 로그 분석
- 파일 단위 조사
- 근거 점검
- 통계 검토
- 문장 정합성 검토
- 관련연구 갭 분석

여러 서브에이전트가 같은 최종 원고나 같은 최종 UX 경로를 동시에 수정하게 두지 않는다.
최종 종합과 최종 쓰기 결정은 하나의 오케스트레이터가 맡는다.

## 원고 관련 작업의 완료 기준
다음이 모두 충족되기 전까지 원고 관련 작업은 끝난 것이 아니다.
- 필요한 Method 정보가 존재하거나, 없을 경우 명시적으로 unavailable 처리되어 있다
- 비교 주장이 필요한 경우, 결과가 단일 aggregate summary를 넘는 근거로 뒷받침된다
- 주요 주장이 현재 근거 수준에 맞는 표현으로 작성되어 있다
- 내부 정합성 검사를 통과했다
- 재현성 주장이 실제 아티팩트와 연결되어 있다
- 최종 요약에 한계와 남은 불확실성이 설명되어 있다
